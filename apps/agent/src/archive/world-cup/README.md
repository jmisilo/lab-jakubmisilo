# Archived World Cup 2026 Module

This module was archived after the tournament ended. It is preserved as a coherent reference implementation, but it is not connected to the running agent.

Current runtime status:

- The agent does not register the World Cup tools or include their routing instructions.
- The Hono app does not expose `GET /jobs/world-cup/events`.
- The former `world-cup-tracking` skill is outside the active `src/skills` directory and is not loaded.
- The `world_cup_2026_*` Drizzle tables and committed migrations remain managed so historical data is preserved.

If an Upstash QStash polling schedule still targets `/jobs/world-cup/events`, disable or delete it after deploying the archived version. Repository code cannot remove an independently configured dashboard schedule, and the disconnected endpoint will return 404.

## Layout

- `app` contains the former route, tools, schemas, domain services, notification renderer, and tests.
- `infrastructure/api` contains the former tournament provider client and tests.
- `infrastructure/db.ts` contains persistence operations for the retained tables.
- `skill` contains the former loadable agent skill.

The archived code remains under `src`, so typechecking and tests can protect the reference implementation even though production entrypoints do not import it.

## Reconnecting A Future Tournament

Treat restoration as a deliberate product change rather than importing this directory from production code in place. The provider, teams, dates, schemas, table names, and notification behavior may all need updating first.

1. Validate the tournament provider contract and update the schemas, team registry, time zones, and tests.

3. Move `infrastructure/api` to `src/infrastructure/world-cup` and `infrastructure/db.ts` to `src/infrastructure/db/services/world-cup.ts`.

## Historical Design

The original system was polling-based with in-memory processing and was designed for this small personal-agent deployment, not for high scale. Root-level app files owned routing, tools, shared schemas/types, and the canonical team registry. Runtime tracking behavior lived under `tracking/*`, with colocated tests.

## Notifications

Notifications are driven by the QStash-scheduled polling endpoint.

1. QStash calls `GET /jobs/world-cup/events`.
2. `WorldCupRouter` verifies the QStash signature with `QSTASH_CURRENT_SIGNING_KEY` and `QSTASH_NEXT_SIGNING_KEY`.
3. After verification, the router calls `WorldCupPollingService.pollAndDeliver({ bot })`.
4. The poller fetches games from `worldcup26.ir/get/games`.
5. Each API snapshot is compared with the previous DB snapshot in `world_cup_2026_game_snapshots`.
6. `WorldCupEventDetector` detects `kickoff-reminder`, `kickoff`, `goal`, and `game-end` events.
7. Detected events are inserted into `world_cup_2026_detected_events` with a unique `event_key`, so repeated polls do not duplicate events.
8. The poller loads active subscriptions from `world_cup_2026_subscriptions`.
9. Subscriptions are matched by event type, team/match scope, and active status.
10. For each matching subscription, the poller creates a delivery row in `world_cup_2026_event_deliveries` with a unique `delivery_key`.
11. Newly created deliveries are sent through `WorldCupNotificationService`.
12. Notification text is generated by AI using the event payload, recent thread transcript, and memory context.
13. The message is posted to the subscribed thread with Chat SDK via `bot.thread(threadId).post(...)`.
14. The delivery is marked `sent` or `failed`.

Kickoff timing uses `WorldCupTimeService`. The API `local_date` is treated as venue-local wall-clock time based on `stadium_id`, then converted to the effective user timezone. Unknown stadium IDs fail closed instead of assuming a timezone.
