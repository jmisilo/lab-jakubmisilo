# Personal Assistant Design

This note captures the agreed direction for evolving `@labjm/agent` from a showcase Telegram bot into a practical personal assistant. Current implementation is useful context, but not sacred; preserve what helps and redesign boundaries where the product needs it.

## Product Direction

- The assistant is a personal operating center, initially optimized for the owner workflow rather than a generic CEO/email product.
- Telegram remains the only user-facing MVP channel. The local TUI stays useful for development and testing.
- Text input comes first. Voice and files come later once the core memory, retrieval, and scheduling model is solid.
- Responses should feel like a natural conversation with a friend. Avoid raw database dumps, noisy confirmations, or routine "I used memory" disclosures.

## Ownership And Identity

- Postgres is the operational source of truth for chat state, personal context, profiles, and scheduled items.
- The app should support multiple separated user accounts, even while product behavior is optimized for the owner first.
- Use an internal `User` entity as the canonical owner.
- Link Telegram and future channels through `ExternalIdentity` records. Channel identities must not become the core owner ID.
- `UserProfile` is required for operational defaults such as timezone, default location, locale, units, and channel preferences.
- Onboarding should be progressive. Do not push an upfront setup flow; ask for missing profile fields only when needed.
- First implementation should route Telegram messages through internal user resolution before expanding assistant capabilities.

## Knowledge Model

- The central durable context unit is `KnowledgeNode`.
- The user is the implicit root; every stored node belongs to that user's knowledge tree.
- Nodes support arbitrary depth. Five or more nesting levels are expected and should not require a schema change.
- A node can be a leaf today and later gain children. Do not model "directory" and "note" as mutually exclusive permanent types.
- Each node has a title, slug/path segment, body, parent pointer, status, confidence, metadata, embedding, and timestamps.
- Leaf-like nodes usually carry specific knowledge. Internal/group-like nodes usually carry a concise description or synthesis of their subtree.
- Node shape starts flexible, but not shapeless: a small required core plus typed optional metadata by content type.
- Initial item types can include `fact`, `preference`, `person`, `relationship`, `project`, `role`, `decision`, `plan`, `note`, `task`, `habit`, and `event`, with `note` as the fallback for ambiguous items.
- Initial statuses are `active`, `archived`, `superseded`, `planned`, and `uncertain`.
- Confidence is first-class, for example `high`, `medium`, and `low`.
- Titles are required. They can be user-provided or generated by the service.
- Store `body` as concise normalized knowledge text. Preserve exact original wording through source references to chat messages.
- Every durable object and event should have `createdAt`; mutable objects should also have `updatedAt`.
- Current active nodes can be updated in place, with lightweight history through `KnowledgeEvent`.
- Corrections and changed life facts should preserve history. For example, "I work at Company Y" can supersede the active Company X employment node without deleting the Company X job-history context.
- Derived facts are explicit nodes linked to their source facts with uncertainty. Do not silently rewrite derived facts as hard truth.
- Existing `agentNotedMemories` should be replaced by `KnowledgeNode`; do not keep two active durable memory systems.
- There is no production noted-memory data to preserve, so the old `agentNotedMemories` table can be removed immediately.
- Rolling compressed conversation summaries remain separate from curated knowledge.

## Structure And Graph

- The vault-like structure is a reference shape, not an Obsidian integration.
- Use dynamic user-owned `KnowledgeNode` records for hierarchy.
- Nodes are created automatically when classification is clear.
- Nodes have human-readable titles plus stable IDs and path slugs.
- Parent-child links represent the primary tree location. Cross-cutting membership and relationships use controlled `KnowledgeLink` types.
- Initial link types: `related_to`, `part_of`, `derived_from`, `supersedes`, `contradicts`, `supports`, `mentions`, `about_person`, and `about_project`.

## Ingestion

- The conversation orchestration agent owns deciding when to preserve durable context.
- Implicit durable context can be saved silently during the normal agent turn.
- Explicit save/update requests should be handled inline and acknowledged.
- Ingestion should batch the latest user message with recent conversation context, not process isolated messages only.
- MVP ingestion uses user messages only. Assistant outputs are not truth. Tool/research output ingestion comes later and should be selective.
- Knowledge writes should go through a constrained `manage-knowledge` tool with typed actions. Application code validates tool inputs and owns database writes, but the orchestrating agent chooses the mutation.
- Save clear durable context automatically, including sensitive personal context stated by the user. Ask only when context is unclear or an external side effect is involved.
- Store useful derived context with uncertainty. Example: if the user says they are 25 on July 3, 2026, store that they were likely born in 2000 or 2001, not an exact birthday.
- Useful uncertainty should be stored explicitly as uncertain, not discarded.
- Every knowledge write should retain source references to originating chat messages when available.
- Explicit knowledge commands should use a constrained normal-agent tool named by capability, `manage-knowledge`.
- `manage-knowledge` should support create, update, correct, and archive. Hard delete remains explicit/admin-only.

## Knowledge Service Boundary

- Use one shared `KnowledgeService` boundary for agent-driven ingestion, explicit knowledge commands, retrieval, and context assembly.
- `KnowledgeService` should expose outcome-oriented methods such as `manageKnowledge`, `searchRelevantKnowledge`, and `buildKnowledgeContext`.
- Model/tool layers must not know table details.
- `KnowledgeService` owns embedding generation and decides what text to embed and when embeddings should be regenerated.
- If embedding generation fails during a write, save the knowledge without embedding, log the failure, and allow later repair/backfill.

## Retrieval

- Retrieval should support both implicit use inside answers and explicit questions such as "what do you know about my job search?"
- Explicit retrieval synthesizes an answer by default. Raw item dumps are not the product surface.
- Normal answers should use context naturally without announcing memory usage.
- Default retrieval should prioritize active items plus relevant uncertain/planned items. Historical or superseded items are included when the query asks for history or when needed to explain change.
- Implicit context assembly and explicit retrieval should use the same retrieval pipeline with different formatting and limits.
- Retrieval should be hybrid from day one: semantic similarity, status, recency/updatedAt, confidence, and tree relevance.
- Search all nodes, not only leaves. Selected specific nodes should include their ancestor chain for context.
- When a group/internal node scores highly, retrieval may include its most relevant children, but should not dump all descendants by default.
- Sibling and cousin nodes are included only when they independently score well or the user explicitly scopes broadly. This prevents a `projects/project-x/design-system` query from pulling unrelated `project-y` context.
- Tree hierarchy should influence ranking as a soft boost, not a hard filter unless the user explicitly scopes the query.
- `KnowledgeService` should return structured retrieval results plus metadata. `AgentContextService` formats those results into model context.
- Store multilingual content as-is. Do not normalize or translate all knowledge into English.

## First Milestone

- The first implementation milestone is identity plus knowledge foundation only, with no reminders yet.
- Introduce `User`, `ExternalIdentity`, `UserProfile`, `KnowledgeNode`, `KnowledgeLink`, and `KnowledgeEvent`.
- Replace noted-memory creation/retrieval with knowledge-backed behavior.
- Keep `AgentContextService` as the context orchestration layer, but swap durable memory internals to knowledge retrieval.
- Add the explicit `manage-knowledge` tool for user-requested saves and edits.

## Scheduled Items

- Tasks, reminders, habits, and calendar events are operational scheduled items, not `KnowledgeNode`s.
- Use one unified `ScheduledItem` model with typed metadata by scheduled item type.
- Scheduled item content should still be searchable and can be linked to knowledge.
- MVP scheduling means reminders first, not a full todo-list system.
- Natural-language scheduling input is accepted, but the system should compile it into executable schedule data.
- Infer nearest future time from context when reasonable. Use the user's profile timezone, initially `Europe/Warsaw`, unless the request specifies another relevant timezone/location.
- Reminder delivery should happen in the same Telegram thread for MVP.
- Scheduling creates a future side effect, so it should get a concise confirmation with resolved time/timezone.

## QStash Delivery

- Use QStash as the trigger engine, not database polling, for reminders.
- Keep Postgres as source of truth. QStash is a delivery trigger/projection.
- Use one QStash resource per scheduled item:
  - one-time reminders store a QStash `messageId`;
  - recurring reminders store a QStash `scheduleId`.
- Edits should cancel/delete/overwrite/recreate the QStash resource and update Postgres.
- Each execution creates a `ScheduledItemDelivery` record for idempotency, retries, and history.
- QStash should call an internal delivery endpoint with stable IDs. The endpoint loads current DB state before delivery.
- Scheduled delivery should call a separate scheduled-agent runtime with scheduled-specific instructions.
- The scheduled runtime may use the same tool registry initially.
- Reminder output should be generated naturally by the model, not posted as raw stored text.
- If tool use fails for a reminder, send a degraded reminder when possible. For jobs where the tool result is the point, mark failure and optionally notify the user.

## Research

- Simple research remains synchronous and bounded for MVP.
- Longer research can become background jobs later.
- Research outputs should be saved only when they produce durable value or the user asks for preservation.
